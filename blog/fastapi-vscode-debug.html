1. Why the print() Habit Hurts

Every Python developer has done it — hit a bug, scatter a handful of print(some_var) statements across the handler, restart the server, fire a request, squint at the terminal, then do it all over again. It works eventually, but it is slow, noisy, and teaches you nothing about the control flow you did not already suspect.

A proper interactive debugger gives you:

  • Breakpoints — pause execution at any line without touching source code
  • Variable inspection — see the full object tree, not just what you remembered to print
  • Step-through execution — watch exactly which branches your code takes
  • Call-stack navigation — jump between frames to understand how you got here
  • Expression evaluation — run arbitrary code in the paused context

FastAPI + VS Code makes all of this trivially easy once you have a launch.json in place. Here is the exact setup I use in production-like environments.

2. Prerequisites

Make sure the following are ready before proceeding:

  • VS Code with the Python extension (ms-python.python)
  • Python 3.10+ virtual environment activated in your project
  • fastapi and uvicorn[standard] installed in that environment
  • Optional: Docker + Dev Containers extension for section 5
Note All commands below assume your project root contains main.py and your virtual environment lives at .venv/. Adjust paths to match your project structure.

3. Setting Up launch.json

Press Ctrl+Shift+P (or Cmd+Shift+P on Mac) and run Debug: Open launch.json. If no file exists yet, VS Code creates .vscode/launch.json for you. Replace its content with: 

{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "FastAPI: Uvicorn",
      "type": "debugpy",
      "request": "launch",
      "module": "uvicorn",
      "args": [
        "main:app",
        "--reload",
        "--host", "0.0.0.0",
        "--port", "8000"
      ],
      "jinja": true,
      "justMyCode": true
    }
  ]
}

Key fields explained:

  • "module": "uvicorn" — runs python -m uvicorn so debugpy can inject itself. Never use "program" with the uvicorn binary.
  • "--reload" — hot-reload stays active; breakpoints survive file saves.
  • "justMyCode": true — skips stepping into FastAPI / Starlette internals.
Warning Do not set "program": "path/to/uvicorn". That launches the binary directly and debugpy cannot attach. Always use "module": "uvicorn".

4. Running FastAPI Under the Debugger

With launch.json saved, open main.py and click the gutter next to any line inside a route handler. A red dot confirms the breakpoint is set. 

from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class Item(BaseModel):
    name: str
    price: float

@app.post("/items/")
async def create_item(item: Item):
    # ← click the gutter here to set a breakpoint
    result = {"item_name": item.name, "item_price": item.price}
    return result

Press F5 to start. VS Code launches uvicorn and attaches debugpy. Send a request:

curl -X POST http://localhost:8000/items/ \
  -H "Content-Type: application/json" \
  -d '{"name": "widget", "price": 9.99}'

VS Code pauses at your breakpoint. Hover over item to inspect the full Pydantic model, type expressions in the Debug Console, or step through with F10 (step over) and F11 (step into).

Pro tip Open the Watch panel and add expressions like item.model_dump(). They re-evaluate every time execution pauses — a live dashboard for your key values.

5. Debugging Inside Docker

When your FastAPI app runs in a container, expose the debugpy port and tell VS Code to attach remotely.

5.1 — Update your Dockerfile

FROM python:3.11-slim

WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt debugpy

COPY . .

CMD ["python", "-m", "debugpy",
     "--listen", "0.0.0.0:5678",
     "--wait-for-client",
     "-m", "uvicorn", "main:app",
     "--host", "0.0.0.0", "--port", "8000", "--reload"]

5.2 — Expose ports in docker-compose

services:
  api:
    build: .
    ports:
      - "8000:8000"   # FastAPI
      - "5678:5678"   # debugpy
    volumes:
      - .:/app

5.3 — Add a remote-attach config

{
  "name": "FastAPI: Docker Attach",
  "type": "debugpy",
  "request": "attach",
  "connect": { "host": "localhost", "port": 5678 },
  "pathMappings": [
    { "localRoot": "${workspaceFolder}", "remoteRoot": "/app" }
  ],
  "justMyCode": true
}

Run docker compose up. The container waits for a debugger client. Select FastAPI: Docker Attach in VS Code and press F5 — the container unblocks and your breakpoints work exactly as they do locally.

Security reminder Never expose port 5678 in production. Use a separate docker-compose.debug.yml override file. The --wait-for-client flag blocks all traffic until a debugger connects.

6. Tips & Tricks

Conditional breakpoints

Right-click a breakpoint dot → Edit Breakpoint → add a condition like item.price > 100. The debugger pauses only when the condition is true.

Logpoints — zero-code tracing

Right-click the gutter → Add Logpoint. Enter a message like Processing: {item.name}, price={item.price:.2f}. VS Code logs it on every pass without pausing — a structured, toggleable print() that leaves no trace in source.

Debugging background tasks

from fastapi import BackgroundTasks

@app.post("/send-email/")
async def send_email(background_tasks: BackgroundTasks, email: str):
    background_tasks.add_task(process_email, email)
    return {"message": "queued"}

async def process_email(email: str):
    # Breakpoints work here — same event loop, debugpy stays attached
    ...

Exception breakpoints

In the Run & Debug sidebar, expand Breakpoints and check Raised Exceptions. VS Code pauses the instant any exception is raised, before FastAPI's handlers can swallow it.

7. Wrapping Up

The setup here — launch.json with "module": "uvicorn" for local work and debugpy --listen for Docker — covers 95% of the debugging scenarios I encounter day to day. Set it up once, commit it, and your whole team gets it for free.

TL;DR checklist
  • Add .vscode/launch.json with "module": "uvicorn"
  • Use "justMyCode": true to stay focused on your own code
  • For Docker: install debugpy, expose port 5678, use --wait-for-client
  • Try conditional breakpoints and logpoints before reaching for print()
