1. Why the print() Habit Hurts

Every Python developer has been there — hit a bug, scatter print(some_var) statements everywhere, restart the server, fire a request, squint at the terminal, repeat. It works eventually, but it is slow and teaches you nothing about the control flow you didn't already suspect.

A proper interactive debugger gives you:

  • Breakpoints — pause execution at any line without touching source code
  • Variable inspection — see the full object tree, not just what you remembered to print
  • Step-through execution — watch exactly which branches your code takes
  • Call-stack navigation — jump between frames to understand how you got here
  • Expression evaluation — run arbitrary code in the paused context

2. How Debugging Actually Works

Before touching any config, it helps to understand what happens when you press F5 in VS Code. A lot of "breakpoints not hitting" problems come from one of these five steps breaking.

1
VS Code reads launch.json
Picks your virtualenv Python, your working directory, and your args
↓
2
debugpy starts
Wraps your Python process so VS Code can pause and inspect it
↓
3
uvicorn launches as a module
python -m uvicorn main:app — the -m flag is what lets debugpy attach
↓
4
Your FastAPI app starts
Routes register, startup events fire, server listens on port 8000
↓
5
Request hits a breakpoint → VS Code pauses
You can now inspect variables, step through code, evaluate expressions
Key insight If debugging fails, it's almost always step 1 (wrong interpreter or cwd), step 3 (using "program" instead of "module"), or a PYTHONPATH mismatch. The fixes in section 7 map directly to these steps.

3. Prerequisites

Make sure the following are ready before proceeding:

  • VS Code with the Python extension (ms-python.python)
  • Python 3.10+ virtual environment activated in your project
  • fastapi and uvicorn[standard] installed in that environment
  • Optional: Docker + Dev Containers extension for section 6

Real-world project structure

Most tutorials show a single main.py. Real repos look more like this — and the debug config needs to account for it: 

my-api/
├── .venv/                  ← your virtualenv
├── .vscode/
│   └── launch.json         ← debug config lives here
├── app/
│   ├── __init__.py
│   ├── main.py             ← FastAPI app instance
│   ├── routers/
│   │   └── orders.py
│   └── models.py
├── requirements.txt
└── .env
Note All commands below assume this structure. The cwd and args fields in launch.json will need to match your actual layout.

4. Setting Up launch.json

Press Ctrl+Shift+P (or Cmd+Shift+P on Mac) and run Debug: Open launch.json. VS Code creates .vscode/launch.json if it doesn't exist. Paste this: 

{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "FastAPI: Uvicorn",
      "type": "debugpy",
      "request": "launch",
      "module": "uvicorn",
      "args": [
        "app.main:app",
        "--reload",
        "--host", "0.0.0.0",
        "--port", "8000"
      ],
      "cwd": "${workspaceFolder}",
      "envFile": "${workspaceFolder}/.env",
      "env": {
        "PYTHONPATH": "${workspaceFolder}"
      },
      "jinja": true,
      "justMyCode": true
    }
  ]
}

Understanding app.main:app — the most common confusion

The string "app.main:app" follows this pattern: 

"package.module:variable"
      ↑         ↑        ↑
  folder    filename   FastAPI()
  (app/)  (main.py)   instance name

Examples for common structures:

  • main.py at root with app = FastAPI() → use "main:app"
  • app/main.py with app = FastAPI() → use "app.main:app"
  • src/server.py with api = FastAPI() → use "src.server:api"

Key fields explained

  • "module": "uvicorn" — runs python -m uvicorn so debugpy can inject itself. Never use "program" — it launches the binary and debugpy cannot attach.
  • "cwd" — sets the working directory Python imports resolve from. Wrong cwd = "Could not import module" errors.
  • "PYTHONPATH" — adds your project root to Python's module search path. Essential when your app is in a subdirectory.
  • "envFile" — loads .env variables automatically so your app config works the same as in production.
  • "justMyCode": true — skips stepping into FastAPI / Starlette internals.
Warning Do not set "program": "path/to/uvicorn". That launches the binary directly and debugpy cannot attach. Always use "module": "uvicorn".

5. Running FastAPI Under the Debugger

With launch.json saved, open any route handler file and click the gutter next to a line number. A red dot confirms the breakpoint is set. 

from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class Item(BaseModel):
    name: str
    price: float

@app.post("/items/")
async def create_item(item: Item):
    # ← click the gutter here to set a breakpoint
    result = {"item_name": item.name, "item_price": item.price}
    return result

Press F5 to start. VS Code launches uvicorn and attaches debugpy. Send a request:

curl -X POST http://localhost:8000/items/ \
  -H "Content-Type: application/json" \
  -d '{"name": "widget", "price": 9.99}'

VS Code pauses at your breakpoint. You can now:

  • Hover over item to see the full Pydantic model
  • Open Debug Console and type item.price * 1.1
  • Press F10 to step over a line, F11 to step into a function call
  • Press F5 again to continue to the next breakpoint
Pro tip Open the Watch panel (Run & Debug sidebar) and add expressions like item.model_dump(). They re-evaluate every time execution pauses — a live dashboard for your key values.

6. Debugging Inside Docker

When your FastAPI app runs in a container, expose the debugpy port and tell VS Code to attach remotely.

6.1 — Update your Dockerfile

FROM python:3.11-slim

WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt debugpy

COPY . .

# debugpy listens on 5678; uvicorn serves on 8000
CMD ["python", "-m", "debugpy",
     "--listen", "0.0.0.0:5678",
     "--wait-for-client",
     "-m", "uvicorn", "app.main:app",
     "--host", "0.0.0.0", "--port", "8000", "--reload"]

6.2 — Expose ports in docker-compose

services:
  api:
    build: .
    ports:
      - "8000:8000"   # FastAPI
      - "5678:5678"   # debugpy
    volumes:
      - .:/app        # live-reload picks up file changes

6.3 — Add a remote-attach config

{
  "name": "FastAPI: Docker Attach",
  "type": "debugpy",
  "request": "attach",
  "connect": { "host": "localhost", "port": 5678 },
  "pathMappings": [
    { "localRoot": "${workspaceFolder}", "remoteRoot": "/app" }
  ],
  "justMyCode": true
}

Run docker compose up. The container blocks at startup (because of --wait-for-client). Select FastAPI: Docker Attach in VS Code and press F5 — the container unblocks and your breakpoints work exactly as they do locally.

Security reminder Never expose port 5678 in production. Use a separate docker-compose.debug.yml override file. The --wait-for-client flag blocks all traffic until a debugger connects.

7. Common Errors & Step-by-Step Fixes

These are the errors people actually Google. Each one maps back to one of the five debug flow steps from section 2.

❌ Could not import module

ERROR: Could not import module "app.main"

Fix checklist — try in order:

  • Check args in launch.json — does "app.main:app" match your actual file path?
  • Check cwd — it should be ${workspaceFolder} (your project root)
  • Add "PYTHONPATH": "${workspaceFolder}" to the env block
  • Make sure app/__init__.py exists (makes the folder a Python package)

❌ Breakpoints not hitting

Breakpoint set but never paused — request goes through without stopping

Fix checklist:

  • Confirm you used "module": "uvicorn" not "program"
  • Check VS Code is using the virtualenv interpreter, not system Python (Ctrl+Shift+P → Python: Select Interpreter → pick the one in .venv/)
  • If --reload is on, the reloader spawns a child process — try disabling it temporarily to confirm breakpoints work first
  • Check the bottom status bar — if it says ⚡ debugpy the debugger is attached

❌ Wrong interpreter / module not found

ModuleNotFoundError: No module named 'fastapi'

Fix checklist:

  • VS Code is using system Python instead of your .venv
  • Select the correct interpreter: Ctrl+Shift+P → Python: Select Interpreter → choose ./.venv/bin/python
  • Or explicitly set it in launch.json: "python": "${workspaceFolder}/.venv/bin/python"

❌ Port already in use

ERROR: [Errno 48] Address already in use

Fix:

# Find the process using port 8000
lsof -i :8000

# Kill it (replace PID with actual number from above)
kill -9 <PID>

8. Tips & Tricks

Conditional breakpoints

Right-click a breakpoint dot → Edit Breakpoint → add a condition like item.price > 100. The debugger pauses only when the condition is true — invaluable inside tight loops or high-traffic endpoints.

Logpoints — zero-code tracing

Right-click the gutter → Add Logpoint. Enter a message like Processing: {item.name}, price={item.price:.2f}. VS Code logs it on every pass without pausing — a structured, toggleable print() that leaves no trace in source code.

Debugging background tasks

from fastapi import BackgroundTasks

@app.post("/send-email/")
async def send_email(background_tasks: BackgroundTasks, email: str):
    background_tasks.add_task(process_email, email)
    return {"message": "queued"}

async def process_email(email: str):
    # Breakpoints work here — same event loop, debugpy stays attached
    ...

Exception breakpoints

In the Run & Debug sidebar, expand Breakpoints at the bottom and check Raised Exceptions. VS Code pauses the instant any exception is raised — before FastAPI's handlers can swallow it — so you can inspect the exact state that triggered it.

9. Wrapping Up

The setup here covers 95% of the debugging scenarios I encounter day to day. Set it up once, commit .vscode/launch.json to your repo, and your whole team gets it for free.

The mental model is the key takeaway: if debugging fails, one of the five steps broke. Use the error checklist in section 7 to pinpoint which one and fix it in under a minute.

Full checklist
  • Use "module": "uvicorn", never "program"
  • Set "cwd": "${workspaceFolder}" and "PYTHONPATH"
  • Use "module:variable" syntax matching your actual file structure
  • Select the virtualenv interpreter, not system Python
  • For Docker: install debugpy, expose port 5678, use --wait-for-client
  • Try conditional breakpoints and logpoints before reaching for print()
