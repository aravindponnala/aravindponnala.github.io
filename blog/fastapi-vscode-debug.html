1. Why the print() Habit Hurts

Every Python developer has done it — hit a bug, scatter a handful of print(some_var) statements across the handler, restart the server, fire a request, squint at the terminal, then do it all over again. It works eventually, but it is slow, noisy, and teaches you nothing about the control flow you did not already suspect.

A proper interactive debugger gives you:

  • Breakpoints — pause execution at any line without touching source code
  • Variable inspection — see the full object tree, not just what you remembered to print
  • Step-through execution — watch exactly which branches your code takes
  • Call-stack navigation — jump between frames to understand how you got here
  • Expression evaluation — run arbitrary code in the paused context

FastAPI + VS Code makes all of this trivially easy once you have a launch.json in place. Here is the exact setup I use in production-like environments.

2. Prerequisites

Make sure the following are ready before proceeding:

  • VS Code with the Python extension (ms-python.python)
  • Python 3.10+ virtual environment activated in your project
  • fastapi and uvicorn[standard] installed in that environment
  • Optional: Docker + Dev Containers extension for section 5
Note All commands below assume your project root contains main.py and your virtual environment lives at .venv/. Adjust paths to match your project structure.

3. Setting Up launch.json

Press Ctrl+Shift+P (or Cmd+Shift+P on Mac) and run Debug: Open launch.json. If no file exists yet, VS Code creates .vscode/launch.json for you. Replace its content with: 

{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "FastAPI: Uvicorn",
      "type": "debugpy",
      "request": "launch",
      "module": "uvicorn",
      "args": [
        "main:app",
        "--reload",
        "--host", "0.0.0.0",
        "--port", "8000"
      ],
      "jinja": true,
      "justMyCode": true
    }
  ]
}

Key fields explained:

  • "module": "uvicorn" — runs python -m uvicorn so debugpy can inject itself into the process. Never use "program" with the uvicorn binary.
  • "--reload" — hot-reload stays active; breakpoints survive file saves.
  • "justMyCode": true — skips stepping into FastAPI / Starlette internals unless you explicitly need to.
Warning Do not set "program": "path/to/uvicorn". That launches the uvicorn binary directly and debugpy cannot attach. Always use "module": "uvicorn".

4. Running FastAPI Under the Debugger

With launch.json saved, open main.py and click the gutter to the left of any line inside a route handler. A red dot confirms the breakpoint is set. 

from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class Item(BaseModel):
    name: str
    price: float

@app.post("/items/")
async def create_item(item: Item):
    # ← click the gutter here to set a breakpoint
    result = {"item_name": item.name, "item_price": item.price}
    return result

Press F5 to start. VS Code launches uvicorn and attaches debugpy. Send a request: 

curl -X POST http://localhost:8000/items/ \
  -H "Content-Type: application/json" \
  -d '{"name": "widget", "price": 9.99}'

VS Code pauses at your breakpoint. Hover over item to inspect the full Pydantic model, type expressions in the Debug Console (try item.price * 1.1), or step through the function with F10 (step over) and F11 (step into).

Pro tip Open the Watch panel and add expressions like item.model_dump(). They re-evaluate every time execution pauses — a live dashboard for your key values.

5. Debugging Inside Docker

When your FastAPI app runs in a container (common in microservice stacks), expose the debugpy port and tell VS Code to attach remotely.

5.1 — Update your Dockerfile

FROM python:3.11-slim

WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt debugpy

COPY . .

# debugpy listens on 5678; uvicorn serves on 8000
CMD ["python", "-m", "debugpy",
     "--listen", "0.0.0.0:5678",
     "--wait-for-client",
     "-m", "uvicorn", "main:app",
     "--host", "0.0.0.0", "--port", "8000", "--reload"]

5.2 — Expose ports in docker-compose

services:
  api:
    build: .
    ports:
      - "8000:8000"   # FastAPI
      - "5678:5678"   # debugpy
    volumes:
      - .:/app        # live-reload picks up file changes

5.3 — Add a remote-attach config

Add this second entry inside the configurations array of your launch.json:

{
  "name": "FastAPI: Docker Attach",
  "type": "debugpy",
  "request": "attach",
  "connect": {
    "host": "localhost",
    "port": 5678
  },
  "pathMappings": [
    {
      "localRoot": "${workspaceFolder}",
      "remoteRoot": "/app"
    }
  ],
  "justMyCode": true
}

Run docker compose up. The container blocks at startup waiting for a client (because of --wait-for-client). Select FastAPI: Docker Attach in VS Code and press F5. The container unblocks, uvicorn starts, and your breakpoints work exactly as they do locally.

Security reminder Never expose port 5678 in production images. Use a separate docker-compose.debug.yml override rather than modifying your production compose file. The --wait-for-client flag blocks all traffic until a debugger connects.

6. Tips & Tricks

Conditional breakpoints

Right-click a breakpoint dot → Edit Breakpoint → enter a condition like item.price > 100. The debugger only pauses when the condition is true — invaluable inside tight loops or high-traffic endpoints where stopping on every call would be impractical.

Logpoints — zero-code tracing

Right-click the gutter → Add Logpoint. Enter a message such as Processing: {item.name}, price={item.price:.2f}. VS Code logs it to the Debug Console on every pass without pausing execution. Think of it as a structured, toggleable print() that leaves no trace in your source code.

Debugging background tasks

from fastapi import BackgroundTasks

@app.post("/send-email/")
async def send_email(background_tasks: BackgroundTasks, email: str):
    background_tasks.add_task(process_email, email)
    return {"message": "queued"}

async def process_email(email: str):
    # Breakpoints work here — FastAPI runs background tasks
    # in the same event loop, so debugpy stays attached
    ...

Exception breakpoints

In the Run & Debug sidebar, scroll to the Breakpoints section at the bottom and check Raised Exceptions. VS Code pauses the instant any exception is raised — before FastAPI's exception handlers can swallow it — letting you inspect the exact state that triggered the error.

7. Wrapping Up

The print() habit is understandable — it is fast and always available. But the moment your FastAPI app grows beyond a handful of routes, or you need to debug an async race condition, or you are chasing a Pydantic validation error three stack frames deep, an interactive debugger pays for itself immediately.

The setup described here — launch.json with "module": "uvicorn" for local work and a debugpy --listen CMD for Docker — covers 95% of the scenarios I encounter day to day. Set it up once, commit it, and your entire team gets it for free.

TL;DR checklist
  • Add .vscode/launch.json with "module": "uvicorn"
  • Use "justMyCode": true to stay focused on your own code
  • For Docker: install debugpy, expose port 5678, use --wait-for-client
  • Try conditional breakpoints and logpoints before reaching for print()
